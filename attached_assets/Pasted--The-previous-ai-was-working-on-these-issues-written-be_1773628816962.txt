🔴 Critical Bugs
1. Dashboard "Sign Lease" button breaks the signing flow In apply/dashboard.html (line 433), the "Review & Sign Lease" link points to:
lease.html?id=APP_ID

…but it's missing the ?token= parameter. Every new lease generated has a tenant_sign_token stored in the DB. When the tenant navigates from the dashboard instead of clicking the email link, they arrive without a token — and the sign-lease Edge Function will reject their signature with "Invalid signing link. Please use the link sent to your email address." Tenants who lose or delete their email are stuck.
Fix: The dashboard needs to either include the token in the link (requires exposing it safely via get_application_status) or the token check should be relaxed for users who arrive via direct app ID lookup.

2. sign_lease_co_applicant doesn't check void/expiry In SCHEMA.sql (lines 768–792), the sign_lease_co_applicant DB function has no checks for:
* lease_status = 'voided'
* Expiry date
But sign_lease (the primary version, lines 719–760) does check both. A co-applicant can successfully sign a voided or expired lease right now.

3. get_lease_financials RPC not defined in SCHEMA.sql The function is called in both lease.html and dashboard.html, but it's not defined anywhere in the main schema. It likely lives in SECURITY-PATCHES.sql — which means on a fresh database deployment, both the tenant signing page and the dashboard will fail to load financial data entirely.

4. tenant_sign_token and applicant_user_idcolumns not in SCHEMA.sql Both columns are used in Edge Functions (generate-lease, sign-lease, process-application) but aren't defined in the main schema. They appear to only exist if SECURITY-PATCHES.sql has been run — a fresh deployment will crash on these column references.

🟠 Security Issues
5. IP address is fetched client-side from external APIs In lease.html (lines 376–386), the client fetches the signing IP from ipify.org, my-ip.io, and ipapi.co as a 3-provider fallback. Problems:
* Spoofable: Anyone can use a VPN or proxy to fake their IP for the signing record
* Rate-limited: ipapi.co free tier has strict rate limits
* Slow: 3 sequential external network calls before signing completes
* Falls back to 0.0.0.0 on failure, which would be stored as a fake audit record
Fix: The sign-lease Edge Function already receives the request — it should extract the real IP from x-forwarded-for server-side and ignore the client-supplied value entirely.

6. No validation that applicant status is approved before lease generation The generate-lease Edge Function never checks app.status. An admin can accidentally send a lease to a pending or even denied applicant. There's nothing blocking it.

7. Lease HTML stored without XSS escaping In sign-lease Edge Function, the buildLeaseHTML()function inserts raw DB values directly into HTML (e.g., app.first_name, app.last_name, app.property_address, app.tenant_signature). If an applicant submitted HTML characters in their name field during application (e.g. <b>Bob</b>), those would render unescaped in the stored lease document.

8. Signature name is never matched against the applicant's name The doSign() function accepts any text as a legal signature — there's no check that it matches or resembles app.first_name + ' ' + app.last_name. Someone with the signing link could type a completely different name. A warning (or soft match) would improve integrity.

🟡 UX / Flow Issues
9. Co-applicant receives two confusing emails When a lease is generated, the co-applicant immediately gets an email (from generate-lease). But when they open their link, they see "Waiting for Primary Applicant." Then when the primary signs, they get a second email telling them to now sign. The initial email creates confusion because the action isn't ready yet. The first co-applicant email should ideally be sent only once the primary has signed.
